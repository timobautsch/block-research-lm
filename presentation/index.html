The core idea behind NotebookLM is simple but powerful: it answers only from the sources you upload — not the open web, not the model's memory. If your documents don't contain the answer, a good system says so instead of making something up. That's what crushes hallucinations versus a normal chatbot.

NotebookLM is upload-first: you add sources before asking anything. It indexes just those materials, then uses retrieval-augmented generation (RAG) to pull the most relevant passages and feed them to Google's Gemini model. The answer comes back with clickable inline citations. Important distinction: Google's product runs on Gemini — ours, as you'll see, deliberately does not.

Ingestion handles ten source types, and the trick is everything normalizes to Markdown. PDFs use the unpdf library; images are OCR'd and described by OpenAI gpt-5.5 vision; audio and video are transcribed by Deepgram Nova-3 after ffmpeg normalizes them; YouTube transcripts come from yt-dlp; URLs are cleaned to markdown or crawled. Each becomes the same kind of Source Block — stable IDs, heading context — so the rest of the pipeline never cares where it came from.

We chunk blocks into ~650-token pieces with a small overlap so context isn't lost at the seams. Each chunk becomes an embedding. In our live config the active embedder is a local ONNX model — multilingual e5-small, 384-dim — running right on the machine, free and offline. Voyage voyage-3-large and OpenAI embeddings are wired in as cloud fallbacks. The vectors are stored durably in Supabase pgvector for fast similarity search.

Retrieval is hybrid, not just vector search. We rewrite the query into variants, then score every candidate chunk on five signals: keyword overlap, BM25, dense vector similarity, query-rewrite overlap, and a heading/entity match. We diversify with an MMR-style step to kill redundancy, then assemble an Evidence Pack — eight items for chat, twelve for artifacts, eighteen for audio — each with an ID like E1 and an exact quote.

This is the trust story. Claude is prompted to answer only from the Evidence Pack, every factual sentence ending in a citation. But we don't just trust the model — a Citation Verifier splits the answer into claims and scores each against the quote it cited. Unsupported claims are stripped, the whole audit lands in a persisted Citation Ledger, and if everything fails the system abstains. That's how we guarantee the answer is genuinely grounded.

Here's the part everyone asks about: which APIs and why. We picked a best-in-class provider for each distinct job — Claude opus-4-8 for grounded reasoning, a local e5 model for embeddings (Voyage as the cloud fallback), Supabase pgvector for storage, OpenAI for vision and image generation, Deepgram nova-3 for speech-to-text, and ElevenLabs for the two-host audio. Every one is called over plain HTTP with no vendor SDK, and every one has a deterministic local fallback so the app never hard-fails.

A panel asks why, not just what. Five decisions I'd defend: local e5 embeddings over hosted — offline, free, EU-local; the trade-off is a lower quality ceiling, so Voyage is one flag away. Supabase pgvector over a dedicated vector DB — one EU-hosted Postgres for rows and vectors. Plain HTTP over vendor SDKs — no lock-in and a single uniform fallback path. A deterministic token-overlap verifier over an LLM-judge — fast, free and unit-testable, at the cost of being a lexical proxy that can miss paraphrase, which I document. And Claude for generation because it adhered best to grounded JSON and abstention — but the router keeps Gemini, GPT and a local model swappable, so it's not a lock-in.

Deployment: the whole thing ships as a Node 24 container on Google Cloud Run, fronted by Firebase Hosting at the edge. One gotcha — Firebase strips cookies that aren't named __session, so our auth uses exactly that. ffmpeg and yt-dlp are baked into the image. And thanks to those local fallbacks, the same codebase runs end-to-end even with zero API keys — just local models instead of the cloud providers.

Where does this go next? Right now, video and audio are handled by Deepgram transcription — the system hears your sources. The big next step is making it watch them too: sample frames from a video and run GPT vision over each one, so it understands charts, slides, and what's actually on screen — fused with the transcript on a single timeline. That frame analysis is genuinely compute-heavy — overkill for a small notebook — so it would live behind an Ultra / Pro tier, while the light local path stays the free default. The payoff: drop in real video as a source and search what was shown, not just what was said.
